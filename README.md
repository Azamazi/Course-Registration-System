# University Course Registration System

A complete relational database + interactive front end for managing student
course registration — built as a portfolio case study for SQL / database
design work.

## What's included

| File | Purpose |
|---|---|
| `schema.sql` | Full DDL — 7 tables, 2 views, 2 triggers, indexes |
| `seed_data.sql` | Realistic sample data (students, instructors, courses, sections, enrollments) |
| `course_registration.db` | Pre-built SQLite database (schema + seed data already loaded) |
| `course_registration_interface.html` | **Working interface** — open this file in any browser, no server or install needed |

## Open the interface

Just double-click `course_registration_interface.html`, or drag it into a
browser tab. It runs SQLite compiled to WebAssembly (sql.js) entirely
client-side and loads the real `.db` file's data. There's nothing to install
and no backend to stand up — which makes it easy to demo live or record a
screen-capture for a portfolio.

It has four tabs:

1. **Catalog** — every course section on offer, with live seat counts.
2. **Register** — pick a student, register or drop them from a section.
   Try registering a student into a full section, or into a course whose
   prerequisite they haven't completed — the database itself rejects both,
   via triggers, not client-side JavaScript.
3. **My Schedule** — a student's full history: completed, registered, and
   dropped courses, with grades.
4. **Reports** — aggregate stats built with `GROUP BY` / `JOIN` queries:
   enrollment by department, teaching load by instructor.

Registration actions modify the in-memory copy of the database for that
browser session only; reload the page to reset to the seeded dataset.

## Schema design

```
Departments ──< Instructors
     │                │
     │                └──< Sections >── Courses ──< Prerequisites (self-referencing)
     │                          │            │
     └──< Students              └──< Enrollments >──┘
```

- **Departments** — top-level academic unit.
- **Instructors** — belongs to a department.
- **Students** — has a declared major (department), roll number, GPA.
- **Courses** — the catalog entry (e.g. "CS301 – Database Systems").
- **Prerequisites** — self-referencing many-to-many on `Courses`, so any
  course can require any number of others.
- **Sections** — a specific offering of a course in a term, with an
  instructor, room, schedule, and seat capacity.
- **Enrollments** — the actual registration record linking a student to a
  section, with status (`Registered` / `Dropped` / `Completed`) and grade.

### Business rules enforced at the database layer

- `trg_prevent_overbook` — blocks an `INSERT` into `Enrollments` once a
  section's registered count reaches its capacity.
- `trg_check_prereq` — blocks registration into a course if the student
  hasn't completed its prerequisite(s).
- `UNIQUE(student_id, section_id)` — a student can't be registered twice
  into the same section.
- `CHECK` constraints on GPA range, credit range, grade values, and
  enrollment status.

Two views (`v_section_availability`, `v_student_schedule`) pre-join the
common lookups so the front end (and any reporting tool) can query a single
flat table instead of repeating five-way joins.

## Porting to Oracle / MySQL

This was built and tested on SQLite for portability (single file, no server
to install, works instantly in any browser via sql.js). To port to Oracle:

- `INTEGER PRIMARY KEY AUTOINCREMENT` → `NUMBER GENERATED BY DEFAULT AS
  IDENTITY` (or a sequence + trigger on older Oracle versions)
- `TEXT` → `VARCHAR2(n)`
- `REAL` → `NUMBER`
- `date('now')` → `SYSDATE`
- Trigger syntax (`RAISE(ABORT, ...)`) → `RAISE_APPLICATION_ERROR(-20001,
  '...')` inside a `BEFORE INSERT` trigger with `:NEW`/`:OLD` bind
  variables instead of `NEW`/`OLD`.

The table/column design and all constraints carry over unchanged.

## Testing performed

Both triggers were verified directly against the database (see project
notes): filling a 2-seat section and confirming a 3rd registration is
rejected with `Section is full`, and confirming a student without a
completed prerequisite is rejected with `Prerequisite not completed`.
